MUST Fix — Critical Issues (Do These First)

Security holes, data loss risks, and production-breaking problems
01
🔴 MUST
Add LockService to Receipt Counter — Prevent Duplicate Receipt IDs
appscript.txt → generateReceiptId()
Right now two admins submitting at the same moment can get the same receipt ID (MNR-2025-0042 issued twice). In real money management, duplicate receipt IDs are a serious audit failure. Fix: wrap the read-increment-write in LockService.getScriptLock().waitLock(5000) so only one thread runs at a time. Always release the lock in a finally block.
02
🔴 MUST
Add LockService to Email Quota Counter — Prevent Quota Overage
appscript.txt → canSendEmail() + incrementEmailCount()
Two simultaneous contributions both pass the canSendEmail() check when quota is at 89. Both then send, pushing count to 91. Gmail bans the account for exceeding 100/day. Fix: merge the check and increment into one atomic operation protected by LockService. Check → Increment → Return true — all inside the lock.
03
🔴 MUST
Enable Google 2-Step Verification on Owner Account
myaccount.google.com → Security → 2-Step Verification
The owner Google account controls the entire system — Sheets, Apps Script, Drive backups, Gmail sending. If this account is compromised, all donor data, financial records, and passwords are exposed instantly. This is the single most impactful security action. Takes 3 minutes. Do this today.
04
🔴 MUST
Protect All Sheet Header Rows — Prevent Silent Column Corruption
Google Sheets → Right-click Row 1 → Protect Range → Owner only
If any header cell in CONTRIBUTIONS, USERS, EXPENSES, EVENTS, AUDIT_LOGS etc. is accidentally renamed or deleted (by anyone with sheet access), all data writes go to wrong columns silently. The system won't throw an error — it will just store wrong values forever. Protect row 1 on every sheet: USERS, TYPES, OCCASIONS, EXPENSE_TYPES, CONTRIBUTIONS, EXPENSES, AUDIT_LOGS, GOALS, GALLERY, ANNOUNCEMENT, BROADCAST, SETTINGS, EVENTS, EVENT_EXPENSES.
05
🔴 MUST
Lock Spreadsheet Sharing to Owner Only
Google Sheets → Share → Settings → Restricted
The main spreadsheet must be set to "Restricted — only people you explicitly share with". It should NOT be "Anyone with link can view". All donor names, phone numbers, email addresses, and financial records are in this file. Even view-only access to an outsider is a serious privacy breach. Check this setting immediately.
06
🔴 MUST
Move Logo URL from Hardcoded CFG to SETTINGS Sheet
appscript.txt → CFG.logoEmailUrl → move to SETTINGS sheet row
Currently the email logo URL is hardcoded in the script. If the Google Drive file is moved, renamed, or the link expires, ALL email receipts silently show a broken image. No error is thrown. Move logoEmailUrl to the SETTINGS sheet so Admin can update it from the UI without redeploying the script. Key name: logo_email_url.
07
🔴 MUST
Add Server-Side Verification Before All Write (POST) Operations
appscript.txt → every _add*(), _update*(), _delete*() function
Currently the API_URL is a public URL. Anyone who discovers it can call addContribution(), deleteUser(), or addExpense() directly without being logged in. The client checks session but the server does not. Add a SessionToken field: on login, server generates a 32-char token stored in the USERS sheet. Every POST action receives this token and verifies it matches the stored token for that UserId before executing.
08
🔴 MUST
Fix Backup Folder — Prevent Root Drive Clutter and Backup Failure
appscript.txt → autoBackupSystem()
Backups are going to Drive root folder. After 365 daily backups you will have 365 files in root. Also: if Drive storage is full, backup silently fails with no alert. Fix: (1) Create a dedicated "Mandir_Backups" folder, store its ID in PropertiesService. (2) Move backup there. (3) Auto-delete backups older than 30 days. (4) Add try-catch that emails Admin if backup fails.

⭐ Your 4 suggestions are all correct and important. Here is exactly how each should be implemented.
Y1
⭐ YOUR
Registration — Email Verification (OTP to Email Before Approval)
login.html appscript.txt
Current problem: Anyone can register with a fake email (e.g., abc@xyz.com). Admin cannot know if the email actually belongs to that person.

How to implement:
Step 1 — On registration form submit, before saving to USERS sheet, server sends a 6-digit OTP to the entered email using MailApp.sendEmail().
Step 2 — Store the OTP and expiry (15 minutes) in PropertiesService with key like OTP_emailhash.
Step 3 — Show an "Enter OTP" step in the modal after form submit.
Step 4 — User enters OTP; if it matches, save registration with Status: "Pending" (awaiting Admin approval).
Step 5 — If OTP is wrong or expired, reject and offer Resend option (max 3 resends).

Important: OTP email counts toward the 90/day quota. Each registration uses 1 email. Monitor this if registrations are high.
Y2
⭐ YOUR
Mobile Number — Correct Indian Format Validation Everywhere
login.html admin.html appscript.txt
Current state: Login page already validates /^[6-9]\d{9}$/ — this is CORRECT for Indian mobiles (must start with 6, 7, 8, or 9 and be 10 digits).

What's MISSING:
• Registration form in login.html does NOT validate mobile format before submit
• Admin "Add User" form does NOT validate mobile format
• Admin "Edit User" form does NOT validate mobile format
• appscript.txt server-side does NOT validate mobile format — any string is accepted

Fix needed in ALL 3 places:
Client-side: if (!/^[6-9]\d{9}$/.test(mobile)) { show error }
Server-side in sanitize call: add explicit mobile validation in addUser, registerUser, updateUser functions

Display format suggestion: Show mobile as +91-XXXXX-XXXXX in profile view (but store as 10 digits only)
Y3
⭐ YOUR
Password Strength Meter — Registration + Admin Add User + Change Password
login.html admin.html user.html
Current state: Registration only checks length ≥ 6. Admin Add User has no minimum check. Change Password on user.html has no strength check.

Recommended minimum rules (for a temple system):
• Minimum 8 characters (increase from current 6)
• At least 1 number
• At least 1 letter
• No spaces allowed

Visual strength meter (4 levels):
🔴 Weak (under 8 chars or letters only) → 🟠 Fair (8+ chars) → 🟡 Good (letters + numbers) → 🟢 Strong (8+ chars + numbers + symbols)

Where to add:
• Registration modal in login.html — below password field, show colored bar
• Admin "Add User" modal in admin.html — same bar
• Change Password modal in user.html — same bar

Implementation: Pure JavaScript, no library needed. On password input event, calculate score and update a colored div width (0–100%). Block submit if score is "Weak".

⭐ YOUR
Registration — Complete Flow Audit & Missing Checks
login.html appscript.txt → registerUser()
Current registration checks (login.html line ~615):
✅ Name: required
✅ Mobile: required (but format NOT validated — see Y2)
✅ Email: basic regex check
✅ Password: ≥ 6 chars (needs upgrade to 8 + strength — see Y3)

What is MISSING from registration:

🔴 Duplicate mobile check server-side — two users can register with same number

🟠 No profanity or special char limit on Name field
🟡 Address field is optional — should show placeholder with example
🟡 No terms acceptance checkbox ("I agree to temple data usage policy")

Server-side in registerUser() — add these checks:
• Check USERS sheet for existing same Mobile → reject with "Mobile already registered"
• Check USERS sheet for existing same Email → reject with "Email already registered"
• Validate mobile format server-side as last defense

HIGH Priority Changes — Important for Production Quality

H1
🟠 HIGH
Sync constants.js ↔ appscript CFG — Single Source of Truth
constants.js appscript.txt
Both files have the same data (name, location, receiptPrefix, emailDailyLimit, signatory, etc.) manually duplicated. When you update one, you must remember to update the other. Fix: add a SYNC CHECKLIST comment block at the top of both files listing every field that must match. Alternatively, read CFG values from a SETTINGS sheet row so only one place needs updating (preferred long-term).
H2
🟠 HIGH
Add Retry Button on API Timeout — Instead of Silent Failure
app.js → getData() / postData() timeout handler
When JSONP times out after 20 seconds, the current code just rejects the Promise. The user sees nothing (or a generic error toast). For rural slow networks this happens often. Fix: in the catch block of every loadData() call in admin.html and user.html, show a "⟳ Retry" button that re-calls the same function. Auto-retry once after 3 seconds for non-write operations.

🟠 HIGH

login.html appscript.txt
Currently "Forgot Password" link exists in login.html but does nothing (or contacts admin). Every forgotten password requires Admin to manually reset it. Fix: clicking Forgot Password shows a modal → user enters registered email → server sends 6-digit OTP → user enters OTP + new password → server verifies OTP and updates password hash. Uses 1 email per reset from quota.
H4
🟠 HIGH
Receipt Counter — Auto Switch to 5-Digit Padding After 9999
appscript.txt → generateReceiptId()
Current code pads to 4 digits: MNR-2025-0001 ... MNR-2025-9999. If a temple has more than 9,999 contributions in a year (unlikely now, but possible at festivals with high walk-ins), counter goes to MNR-2025-10000 which breaks the consistent format. Fix: change padStart(4,"0") to padStart(counter > 9999 ? 5 : 4, "0") or simply always use 5 digits (MNR-2025-00001).
