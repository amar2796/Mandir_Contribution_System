All Modules — How Every Section Works

Deep-dive into every feature, what it stores, who uses it, and what it does
🔑
Authentication & Login (login.html)
SHA-256 hashes the password on the browser before sending. The hash is compared against the stored hash in the USERS sheet — plain passwords are never stored or transmitted. Session is kept in sessionStorage so closing tab = logout automatically. Two roles exist: Admin (full control) and User/Member (view + limited actions).
👥
User Management (admin.html → Users tab)
Admin can Add / Edit / Deactivate members. Each member has: UserId, Name, Email, Phone, Role (Admin/User), Status (Active/Inactive), and hashed Password. Password reset by Admin re-hashes and stores. Device info is logged for every login in AUDIT_LOGS sheet.
💰
Contributions (admin.html → Contributions tab)
Core module. Records every donation: Donor (UserId), Amount, Type (monthly/special/etc.), Occasion, ForMonth, Year, PaymentMode (Cash/UPI/Bank), Note, PaymentDate, ReceiptId (auto MNR-YYYY-NNNN). Admin can Add/Edit/Delete/Download PDF Receipt. User can only view their own contributions on user.html. Contributions are stored in CONTRIBUTIONS sheet.
🧾
Receipt System (PDF — jsPDF)
100% client-side PDF generation using jsPDF + jsPDF-autotable. No server involved for PDF creation. Receipt shows: Mandir logo, name, location, receipt number, amount (hero), payment mode, donor name, month/year, type, occasion, note, signatory. Gold-and-maroon branded A5 format. Auto-saved as "Receipt_MNR-YYYY-NNNN.pdf". Sequential counter is stored in Apps Script PropertiesService; resets every January.
📤
Email System (Gmail via Apps Script)
Two automated email types: (1) Auto Receipt — sends HTML email to donor when Admin records a contribution. (2) Monthly End Report — aggregate PDF-style email sent to Admin on last day of month via a time-based trigger. Daily quota: 90 emails (Gmail free tier limit is 100; 10 kept as buffer for OTPs). Quota counter stored in PropertiesService and resets daily at midnight IST. Toggle for both features is stored in SETTINGS sheet (on/off per Admin preference).
💸
Expense Management (admin.html → Expenses tab)
Tracks temple spending. Each expense has: Title, Amount, ExpenseType, ForMonth, Year, Note, PaymentDate, AdminName. Separate from Event Expenses. Admin can Add/Edit/Delete. Summary is visible on Dashboard. Stored in EXPENSES sheet.
🎉
Events & Event Expenses (admin.html → Events tab)
Temple events (Hanuman Jayanti, Ram Navami, etc.) are tracked separately. Each event: EventName, Category, StartDate, EndDate, Budget, Status (Upcoming/Ongoing/Completed), Description. Linked EVENT_EXPENSES sheet records spending per event. Deleting an event cascades and removes all linked event expenses automatically.
🎯
Goals (admin.html → Goals tab)
Fundraising goals (e.g., "New Bell", "Renovation Fund"). Each goal: GoalName, TargetAmount, CurrentAmount, Status (Enabled/Disabled). Dashboard shows progress bars for each active goal. Public view also shows goals for transparency.
📊
Dashboard (dashboard.html)
Admin-only summary view. Shows: This Year Collection, This Year Expense, Current Balance (closing balance of last year in yearRows), Active Member Count, Year-wise Summary Table (year, income, expense, closing balance), and Goals progress. Data is refreshed on load. Yearly summary is computed server-side by getYearlySummary() function.
📢
Announcements & Broadcast (admin.html)
Admin can post announcements visible to all logged-in users. Broadcast sends a bulk email to all active members (uses quota). Stored in ANNOUNCEMENT and BROADCAST sheets respectively.
🖼️
Gallery (admin.html → Gallery tab)
Temple photo gallery. Photos are hosted externally (Google Drive public links or any CDN URL). Admin adds image URL + caption. Stored in GALLERY sheet. Users view on user.html.
📝
Audit Logs (AUDIT_LOGS sheet)
Every significant action (add user, delete contribution, change password, addEvent, etc.) is logged with: Timestamp (IST), Admin/User Name, Action, Details, DeviceInfo. Auto-creates the sheet with bold header if not found. Audit failure is silently caught so it never crashes the main operation.
⚙️
Settings Sheet (SETTINGS)
Key-value store inside Google Sheets for runtime toggles. Keys: auto_receipt (1/0), monthly_report (1/0). Admin can toggle from UI; stored persistently here. Defaults to OFF if sheet missing — prevents accidental email spam after redeploy.
🔄
Backup System (autoBackupSystem)
A time-based trigger in Apps Script calls autoBackupSystem() daily/weekly. It copies the entire spreadsheet to Google Drive root folder with name "Backup_Mandir_YYYY-MM-DD". Requires manual trigger setup from Apps Script editor.
🌐
JSONP Communication Layer (app.js)
Since Google Apps Script deployed as Web App doesn't support CORS natively, the system uses JSONP (dynamic <script> tag injection). getData() for GET, postData() for POST (both JSONP). 20-second timeout per request. Errors from Apps Script HTML fallback pages (cold starts) are silently suppressed to prevent false crash alerts in Firefox/Chrome. Client-side cache (5–10 minutes) deduplicates repeated calls.
📱
User Portal (user.html)
Limited interface for temple members. They can: see their own contributions, download their receipts, view announcements, view gallery, view goals, make donate requests. They cannot see other members' data.

System Pros & Constants — Strengths and Weaknesses

Honest evaluation of what this architecture does well and where it struggles
✅ Strengths (Pros)
  • Zero hosting cost — Google Sheets + Apps Script is completely free
  • No server to maintain, no downtime alerts, no DevOps needed
  • Data always in Google Sheets — easy for non-technical admins to audit directly
  • SHA-256 hashing means plain passwords never stored anywhere
  • Client-side PDF means receipts generate even offline (after page load)
  • Sequential receipt IDs (MNR-YYYY-NNNN) are audit-friendly and human-readable
  • Email quota system prevents accidental Gmail ban from overage
  • Audit log is self-healing — creates its own sheet if missing
  • Automatic backup creates daily Sheets copies — disaster recovery exists
  • JSONP error suppression prevents false alarms during Apps Script cold starts
  • Client-side cache reduces Apps Script quota consumption significantly
  • Constants.js makes rebranding for another temple a 1-file change
  • Settings stored in sheet — toggles survive script redeployment
  • IST (GMT+5:30) is hardcoded throughout — no timezone confusion for Indian users
⚠️ Weaknesses & Constraints
  • Apps Script has a 6-minute execution limit per request — large data exports can timeout
  • JSONP is inherently insecure for sensitive data (no SSL-level payload encryption)
  • No real database — Sheets scanning is O(n) per request, slows after ~5,000 rows
  • Only 100 Gmail sends/day free — large temples will hit this wall quickly
  • Session stored in sessionStorage — user loses session on tab close (usability issue)
  • Single-owner Google account — if account is suspended, entire system goes down
  • No real-time updates — page must be refreshed to see new data from others
  • Apps Script cold starts (first request after inactivity) add 3–8 second delay
  • PropertiesService is per-script, not per-user — receipt counter is global (race condition risk)
  • No offline mode — entirely dependent on Google's uptime
  • PDF is generated client-side with English fonts — Hindi/Devanagari not supported in jsPDF without custom font embedding
  • No payment gateway integration — UPI is shown as QR only, not auto-verified
  • Broadcast email uses quota — sending to 50+ members in one go can exhaust daily limit
  • Admin account takeover = full data access (no 2FA)

Quality & Maintainability Improvements

How to make this system cleaner, more reliable, and easier to hand over
🏗️ Code Structure
  • constants.js is duplicated — CFG object in appscript.txt is a manual mirror of APP object in constants.js. Any change must be done in both places. Solution: add a comment checklist at top of both files listing all fields that must stay in sync.
  • All HTML files share the same app.js — good. But admin.html currently contains both logic and UI. Over time, split into admin-ui.js and admin-logic.js to make future edits less risky.
  • Version comment at top of each file — add a one-line version string like // v2.1 — 2025-08 so you always know which deployed version is running.
  • Test each action manually after any Apps Script deploy — create a simple checklist: Login ✓, Add Contribution ✓, Generate PDF ✓, Send Email ✓. Takes 5 minutes and catches 90% of regressions.
📋 Data Quality Rules
  • Amount field: always validate as positive number server-side (sanitize() already does this for strings — add Number() validation explicitly)
  • Phone numbers: enforce 10-digit format on input
  • Email: validate format before saving to USERS sheet
  • Receipt counter: add a MAX check — if counter > 9999, switch to 5-digit padding automatically
  • Contribution ForMonth: restrict to Jan–Dec only, don't allow free-text months
  • Occasion and Type fields: always use dropdown (already done) — never allow free text to prevent duplicate categories
🔧 Maintainability Practices
  • Keep a CHANGELOG.txt in your Drive folder. Every time you edit the script, write date + what changed. Invaluable when handing to another admin.
  • Store a SETUP.txt explaining: which Google account owns it, how to redeploy, trigger setup steps, how to add a new template/type.
  • If a sheet is deleted accidentally, the getSheetData() function returns [] silently — add a server-side health check action that verifies all sheet names exist.
  • Any time you add a new sheet (e.g., EVENTS was added), add it to SHEETS constant AND verify the header row in that sheet matches what the code expects.

Multi-User Safety — Data Integrity When Multiple Users Are Active

Can two users corrupt each other's data? What are the real risks?
Important Context Google Apps Script has a global concurrency model. Each doGet() / doPost() call runs in its own execution thread but writes to the same shared spreadsheet. This is fine for append operations (like adding a contribution row) but risky for read-modify-write operations (like updating a counter).
✅ Safe Operations (No Data Leak)
  • Each user only sees their own contributions on user.html (UserId filter server-side)
  • Passwords are SHA-256 hashed — even Admin cannot reverse them
  • Sheet append (addContribution, addExpense) is safe — Sheets locks the row during append
  • Admin email and phone visible only to Admin role — user.html never fetches full user list
  • Session token is in browser memory — users cannot access other sessions
  • Audit log records who did what — post-event trace is always available
⚠️ Risk Areas
  • Receipt counter: Two simultaneous contributions = possible duplicate receipt ID. Use LockService to fix.
  • Edit + Delete race: Admin A edits row 15 while Admin B deletes row 15 simultaneously. Apps Script handles row operations sequentially but outcome can be unpredictable. Mitigate with row-level locking or optimistic concurrency check (compare ID before write).
  • Email quota counter: Two requests check canSendEmail() at same time — both see count=89 (below 90), both send. Now count should be 91. Fix with LockService in incrementEmailCount().
  • JSONP has no authentication token: The API_URL is technically public. Anyone who knows the URL can call GET actions. POST actions (add/edit/delete) should validate that the caller is logged in — currently Admin verification relies on client-side session, not server-side token.
Recommended Addition: Server-Side Session Token When Admin logs in, generate a 32-char random token server-side, store it in USERS sheet (SessionToken column) with expiry timestamp. Every POST action must include this token; server verifies it before executing. This prevents URL-replay attacks.

Temple Policy Document (Unregistered Trust)

Even without official registration, these rules protect the temple and its donors
Legal Note Since the temple is not formally registered as a trust or society, there is no legal accountability framework yet. This policy document serves as an internal governance charter that should be printed, signed by all admins, and kept on record. It demonstrates good faith and builds donor trust.
📜 Suggested Internal Policy — Shree Hanuman Mandir, Paliya

1. Data Collection Policy

The mandir system collects the following member information: Name, Phone Number, Email Address, and Contribution History. This information is collected solely for the purpose of managing temple finances and communicating with members. No data is sold, shared with third parties, or used for any commercial purpose.

2. Receipt & Financial Transparency Policy

Every financial contribution above ₹0 shall receive a system-generated receipt with a unique sequential ID (format: MNR-YYYY-NNNN). Receipts are permanent and cannot be deleted once issued. All annual income and expense summaries are accessible to all registered members via the dashboard.

3. Password & Account Security Policy

Passwords are stored in encrypted (hashed) form and cannot be recovered in plain text even by the Administrator. Any Admin who forgets their password must be reset by another Admin. No one is authorized to ask a member for their plain-text password.

4. Fund Usage Policy

All expenses must be recorded in the system with a category and description before payment. Any single expense above ₹5,000 should be approved by at least two Admins. Expense records cannot be deleted — only marked as corrected with a new opposing entry.

5. Backup & Data Retention Policy

The system creates an automatic daily backup of all financial records to Google Drive. Backup copies are retained for a minimum of 3 years. In the event of a system failure, the most recent backup shall be used for restoration.

6. Admin Conduct Policy

Admin accounts must not be shared. Each person who manages the system must have their own Admin account. Admin actions are recorded in an Audit Log. Unauthorized deletion of records is grounds for account revocation. The primary Google account owner (account@gmail.com) must never be used for daily operations — create a separate Admin account.

7. Email Communication Policy

The system sends automated emails for: (a) Contribution receipts, (b) Monthly financial summaries. Members can opt out by informing the Admin. Broadcast emails are limited to temple-related announcements only.

8. Donor Privacy Policy

Individual contribution amounts of one member are not visible to other members. Only Admins can see all contribution data. The public-facing donor count and total collections are displayed but not individual donor details.

9. Unregistered Trust Disclaimer

Shree Hanuman Mandir, Paliya is currently managed by community volunteers and is not a registered charitable trust. Contributions are donations made voluntarily for the religious and community welfare activities of the temple. Donors are advised that such donations currently do not qualify for 80G tax deduction. The temple committee intends to seek formal registration to enable tax benefits in the future.

Running Multiple Temples — What Must Change

If you deploy this system for another temple, or manage 3–4 temples from one admin
Two approaches exist: (A) Independent deployment per temple — safest, cleanest, fully isolated data. (B) Single multi-tenant deployment — complex but sharable admin for chains or organizations managing multiple temples.
🏛️ Option A — Independent Deployment Per Temple (Recommended)

Each temple gets its own: Google Sheet (spreadsheet), Apps Script deployment, HTML files hosted separately. Only constants.js changes:

  • Update: name, shortName, nameHindi, village, district, state, phone, email, receiptPrefix, upiId, accountName, accountNo, ifscCode, bankName, bankBranch, signatory
  • Update: CFG object in appscript.txt with same values
  • Update: API_URL in each HTML file to point to the new Apps Script web app URL
  • Update: logoEmailUrl in appscript CFG to the new temple's logo
Effort: ~30 minutes per new temple. Fully isolated — one temple's data cannot affect another's.
🔗 Option B — Multi-Tenant Single Deployment

Not currently supported. Would require adding a TempleId field to every sheet, every query filter. Major architectural change. Not recommended until formal registration and a dedicated developer is available.

🛡️ How Client Cache Helps Load Balance

The existing 5-minute client-side cache (window._mandirCache) means that if 10 users open the dashboard within the same 5-minute window, only 1 Apps Script call is made (by the first user). The other 9 see cached data. This is the single most impactful optimization already in the codebase.

Without cache: 10 concurrent users → 10 script calls. With 5-min cache: 10 users → 1 script call. 90% reduction.

Total registered users: comfortably up to 200

⛈️ Worst Case Scenario

Conditions: Festival day, 50+ people trying to pay, bulk broadcast sent, slow rural internet, 3,000+ rows already in sheet

Concurrent admin sessions: 2–3 before throttling

Concurrent user sessions: 10–15 before cold-start errors

Simultaneous write operations: 2–3 (race condition risk beyond this)

Email after broadcast: Quota exhausted; 0 remaining

Apps Script response time: 5–15 seconds per call with large data

Risk: Script throttle (429 errors) if >30 requests/minute

Key Insight This system is designed for a village-level temple with 50–150 active members. It performs excellently within this range. For temples with 200+ active members or festivals drawing 500+ visitors, plan a load-management strategy: stagger contribution entry, disable broadcast on high-traffic days, and ensure cache is working.

🛡️ Application-Level Security Checklist

Currently Implemented ✅

  • SHA-256 password hashing (client + server)
  • HTML escaping (escapeHtml) on all output to prevent XSS
  • Input sanitization via sanitize() with maxLen limits
  • Amount parsed as Number() to prevent string injection
  • Audit log on all write operations
  • Session in sessionStorage (tab-scoped, not persistent)
  • Email quota guard prevents spam from script bugs
  • JSONP SyntaxError suppression prevents false crash alerts

Not Yet Implemented ⚠️

  • No server-side session token — API is callable without login verification server-side
  • No rate limiting on login attempts — brute-force of PIN possible
  • No 2FA for Admin accounts
  • No input validation on phone/email format server-side
  • LockService missing on counter increments (race condition)
  • No CORS restriction on API URL (inherent JSONP limitation)
  • No data-at-rest encryption (Sheets content is Google-encrypted but not your own key)
  • No IP allowlisting for admin access (village-only access cannot be enforced)
